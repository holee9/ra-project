# IEC 81001-5-1:2021 수명주기 보안 활동 매핑

## 1. 표준 개요

| 항목 | 내용 |
|---|---|
| 공식 제목 | Health software and health IT systems safety, effectiveness and security – Part 5-1: Security – Activities in the product life cycle |
| 발행일 | 2021-12 (IEC 81001-5-1:2021) |
| EU 조화 버전 | EN IEC 81001-5-1:2022 (CENELEC 채택) |
| Interpretation Sheet | IEC 81001-5-1:2021/ISH1:2025 (2025년 발행, 요건 명확화) |
| 상위 표준 관계 | IEC 62443-4-1 기반 → 의료기기 소프트웨어 특화 적용 |
| 보완 관계 | IEC 62304 (SW 수명주기) + ISO 14971 (Risk Management) 확장 |
| 적용 대상 | Software in a Medical Device(SiMD), Software as a Medical Device(SaMD), Health IT systems |

### 1-1. 핵심 성격

- **프로세스 지향(Process-oriented)**: 특정 기술 솔루션 미지정 — 구조화된 보안 프로세스를 QMS에 내재화 요구
- **64개 추가 사이버보안 요건** 도입 (IEC 62304 대비)
- 개발 착수부터 Post-Market 유지보수까지 전 수명주기 보안 커버
- IEC 62443-4-1과의 매핑 테이블 Annex에 수록 (상호 참조 용이)

### 2-1. 3개 지역 비교 매트릭스

| 항목 | MFDS (한국) | FDA (미국) | MDR/EU |
|---|---|---|---|
| **근거 규정** | 의료기기 사이버보안 허가·심사 가이드라인 (2025-01-10) | Section 524B FFDCA + FDA Final Guidance (June 2025) | MDR (EU) 2017/745 Annex I §§17.2, 17.4 |
| **IEC 81001-5-1 지위** | IMDRF N60·N73과 함께 핵심 참조 표준으로 명시 | SPDF와 높은 중첩 — FDA guidance에서 구현 프레임워크 예시로 인용 | EN IEC 81001-5-1:2022 = "state of the art" 근거, NB 사실상 필수 요구 |

| **SBOM 요구** | 2025.01 N73 가이드라인 → SBOM 제출 요구 | 2025 Final Guidance → SPDX/CycloneDX 형식, NTIA 7개 요소 필수 | MDCG 2019-16 Rev.1 → SBOM 포함 권고 (§17.2 해석) |
| **Post-Market** | 이상사례·취약점 보고 절차 요구 | 지속적 취약점 모니터링·패치·MDR 803 보고 | PMCF/PMS Plan 내 사이버보안 모니터링 포함 |
| **적용 시점** | 현행 인허가 심사 즉시 적용 | 2023-10-01 이후 제출 의무 (Section 524B) | MDR 발효 후 즉시 (2021-05-26~) |

### 2-2. MFDS 구체 요건 (2025-01-10 가이드라인 기반)

MFDS는 IMDRF 가이드라인 3종을 국내 적용 기준으로 채택:
- **N60**: 의료기기 사이버보안 원칙 및 실무 (핵심 원칙)
- **N70**: 레거시 의료기기 사이버보안 원칙 및 실무
- **N73**: SBOM 원칙 및 실무

IEC 81001-5-1은 N60·N73·ISO 14971과 함께 **보안 인허가 핵심 요건**으로 규정. 제조자는 설치·운영·유지보수·폐기 단계까지 보안 요구사항을 문서화해야 함.

### 2-3. FDA 구체 요건 (June 2025 Final Guidance 기반)

FDA SPDF(Secure Product Development Framework)와 IEC 81001-5-1의 중첩 영역:

| SPDF 요소 | IEC 81001-5-1 대응 조항 |
|---|---|
| Security Risk Management | §5 Security risk management process |
| Threat Modeling | §6.1 Threat analysis and risk assessment |
| Secure Design & Development | §6.2 Secure design / §6.3 Secure implementation |
| Testing & Validation | §6.4 Security verification & validation |
| Post-Market Monitoring | §7 Security maintenance / §7.3 Vulnerability management |
| SBOM | §6.3.5 Software composition / Annex A.4 |

eSTAR 12개 Cybersecurity 문서 중 IEC 81001-5-1 프로세스 기반으로 작성 가능한 항목: Cybersecurity Risk Assessment, SBOM, Security Test Report, Post-Market Surveillance Plan.

## 3. IEC 81001-5-1 수명주기 활동 구조

### 3-1. 5대 보안 프로세스 (Clause 5~9)

```
Clause 5: 일반 보안 프로세스 요건
  └─ 5.1 QMS 내 보안 프로세스 수립
  └─ 5.2 IT Security Risk Management 체계
  └─ 5.3 Component Risk Assessment

Clause 6: SW 개발 수명주기 보안 활동
  └─ 6.1 보안 요구사항 정의
  └─ 6.2 보안 아키텍처 설계
  └─ 6.3 보안 구현 (Secure Coding)
  └─ 6.4 보안 검증 및 검사 (V&V)
  └─ 6.5 보안 릴리스 관리

Clause 7: SW 유지보수 보안 활동
  └─ 7.1 유지보수 계획
  └─ 7.2 문제 해결 프로세스
  └─ 7.3 취약점 관리
  └─ 7.4 보안 패치 배포

Clause 8: 보안 형상 관리
  └─ 8.1 형상 식별 (SBOM 포함)
  └─ 8.2 형상 통제 및 감사

Clause 9: 보안 문제 해결 프로세스
  └─ 9.1 문제 접수 및 분류
  └─ 9.2 취약점 공개 정책 (CVD)
```

### 3-2. IEC 62304 수명주기와의 통합 매핑

| IEC 62304 단계 | IEC 81001-5-1 추가 보안 활동 |
|---|---|
| **5.1 SW 개발 계획** | 보안 계획 수립 (Security Plan), 위협 분석 범위 정의 |
| **5.2 SW 요구사항 분석** | 보안 요구사항 도출 (Threat Model 기반), IDAM 요건 정의 |
| **5.3 SW 아키텍처 설계** | 보안 아키텍처 설계 (최소 권한 원칙, Defense-in-Depth), Attack Surface 분석 |
| **5.4 SW 상세 설계** | Secure Design Patterns 적용, 금지 함수 목록 배제 |
| **5.5 SW 단위 구현** | 안전 코딩 표준 (MISRA C / CERT C/C++) 적용, SAST 도구 실행 |
| **5.6 SW 통합 및 통합 시험** | Fuzz Testing, 취약점 스캔, Penetration Test (독립 평가자) |
| **5.7 SW 시스템 시험** | 종합 보안 V&V, Security Test Report 작성 |
| **5.8 SW 릴리스** | SBOM 확정 (SPDX/CycloneDX), 서명된 릴리스, 안전 업데이트 메커니즘 |
| **6. SW 유지보수** | CVE 모니터링, 패치 프로세스, 취약점 공개 절차 (CVD Policy) |

### 3-3. Annex A 핵심 실무 (규범적 Best Practices)

**A.4 Secure Coding 최소 요건:**
- 알려진 보안 취약 패턴(Development/Design) 회피
- Banned Functions 사용 금지 (gets(), strcpy() 등)
- 미정의·미명세 동작에 의존하는 코드 금지
- 표준 안전 코딩 표준 사용: MISRA C, CERT C/C++
- 자동 정적 분석 도구(SAST) 활용 필수

## 4. 자사 3개 제품 영향 맵핑

### 4-1. X-ray Detector (디지털 평판 검출기)

| 보안 활동 | 적용 내용 | 우선순위 |
|---|---|---|
| Threat Modeling | 디지털 이미지 전송 경로(DICOM/워크리스트) 위협 분석 | 높음 |
| Security Requirements | DICOM TLS, 접근 제어, 감사 로그 요건 | 높음 |
| Secure Architecture | 최소 권한 원칙, 네트워크 격리(Microsegmentation) | 높음 |
| SBOM | 임베디드 SW 컴포넌트 목록 (FPGA 펌웨어 포함) | 중간 |
| V&V (Pen Test) | DICOM 프로토콜 취약점 검증, 물리적 포트 보안 | 높음 |
| Post-Market | 펌웨어 CVE 모니터링, OTA 업데이트 안전성 | 중간 |
| 3지역 적용 | MFDS N60 §3 + FDA eSTAR Cybersecurity + MDR Annex I §17.2 | 전체 |

**Key Risk**: DICOM 네트워크 연결 → 병원 내부망 침투 경로 가능. 네트워크 인터페이스 최소화, TLS 1.2+ 필수.

### 4-2. Handheld X-ray Source (휴대용 X선 발생장치)

| 보안 활동 | 적용 내용 | 우선순위 |
|---|---|---|
| Threat Modeling | 무선 제어(Bluetooth/Wi-Fi) 위협, 물리적 도난·조작 | 높음 |
| Security Requirements | 사용자 인증, 노출 파라미터 조작 방지, 펌웨어 무결성 | 높음 |
| Secure Architecture | Secure Boot, 암호화된 통신, 하드웨어 보안 요소(HSM) | 중간 |
| SBOM | 펌웨어·제어 SW 컴포넌트 목록 | 중간 |
| V&V (Pen Test) | 무선 인터페이스 공격 시뮬레이션, 파라미터 변조 시험 | 높음 |
| Post-Market | 펌웨어 업데이트 채널 보안, 필드 취약점 보고 절차 | 중간 |
| 3지역 적용 | MFDS N60 + FDA 524B (Cyber Device 판정 필요) + MDR §17.4 | 전체 |

**Key Risk**: 무선 연결 시 방사선 조사 파라미터 무단 변경 → 환자 안전 직결. Secure Boot + 암호화 인증 필수.

**주의**: Handheld Source가 Wi-Fi/Bluetooth 연결 기능 미보유 시 FDA "Cyber Device" 해당 가능성 재검토 [firm 연결기능 확인]

### 4-3. 촬영실 GUI SW (독립형 Software as a Medical Device)

| 보안 활동 | 적용 내용 | 우선순위 |
|---|---|---|
| Threat Modeling | HIS/RIS 연동 인터페이스, 사용자 계정 탈취, SQL Injection | 높음 |
| Security Requirements | 역할 기반 접근 제어(RBAC), 세션 관리, 입력값 검증 | 높음 |
| Secure Architecture | Least Privilege 설계, API 보안, 데이터 암호화(저장·전송) | 높음 |
| SBOM | 사용 OSS 라이브러리 전체 목록 (CVE 연계 필수) | 높음 |
| Secure Coding | CERT C/C++ 또는 OWASP Top 10 대응 코딩 표준 | 높음 |
| V&V | DAST + SAST + Pen Test, 독립 평가자 수행 | 높음 |
| Post-Market | CVE 피드 자동 모니터링, 보안 패치 SLA 정의 | 높음 |
| 3지역 적용 | MFDS 디지털의료기기 가이드라인 + FDA SaMD + MDR SaMD | 전체 |

**Key Risk**: SaMD로서 가장 넓은 공격 표면. IEC 81001-5-1 전 조항 적용 우선순위 최고.

## 5. 구현 요건 체크리스트 (자사 적용용)

### 5-1. QMS 통합 필수 활동

- [ ] **Security Plan** 수립 — SW 개발 계획(IEC 62304 §5.1)에 보안 계획 병합
- [ ] **Threat Model** 문서 작성 — STRIDE 기반, 제품별 Attack Surface 분석 (→ `FDA_Threat_Model_STRIDE_가이드.md` 참조)
- [ ] **Security Risk Assessment** — ISO 14971 Risk Management File에 사이버보안 위험 통합 (별도 트랙 금지)
- [ ] **Security Requirements Spec** — SRS에 보안 요건 섹션 추가
- [ ] **Secure Coding Standard** 선택·적용 — MISRA C (펌웨어) / CERT C/C++ (SW)
- [ ] **SBOM 생성** — SPDX 또는 CycloneDX 형식 (→ `SBOM_생성_실무.md` 참조)
- [ ] **Penetration Test** 계획·수행 — 독립 평가자 참여
- [ ] **CVD Policy** 수립 — 취약점 접수·분류·공개 절차 문서화
- [ ] **Post-Market Security Plan** — CVE 모니터링 주기, 패치 SLA, 사용자 통지 절차

## 7. 요약

IEC 81001-5-1:2021은 의료 헬스 소프트웨어의 **전 수명주기 사이버보안 프로세스 표준**으로, IEC 62304 소프트웨어 수명주기에 64개 보안 활동을 추가하는 구조다. 3개 지역 모두에서 사실상 필수 표준으로 인정되며, EN IEC 81001-5-1:2022는 MDR NB의 "state of the art" 근거이고, FDA 2025 Final Guidance는 SPDF 구현 프레임워크로 명시 인용한다. MFDS는 2025-01-10 가이드라인에서 N60·N73과 함께 핵심 참조 표준으로 규정했다.

자사 3개 제품 모두 IEC 81001-5-1 적용 대상이며, GUI SW가 가장 넓은 적용 범위(전 조항)를 가지고, Handheld Source는 Secure Boot·무선 보안에 집중, X-ray Detector는 DICOM 네트워크 보안에 집중한다.
